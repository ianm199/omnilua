# 5.3 follow-up: string→number coercion in core ops + arith error wording

DISCOVER packet `coerce_err`. Read-only investigation; nothing edited. Every
repro below was reproduced against `lua5.3.6` via
`specs/oracle/diff_one.sh 5.3 '<code>'` (and cross-checked on 5.4/5.5). The
oracle contract (`specs/oracle/CONTRACT.md`): default `make macosx` builds,
`LUA_COMPAT_MATHLIB` on for 5.3.

There are **two** divergence categories, both rooted in the same architectural
fact: our shared core implements the **5.4** model where string operands never
coerce in core arithmetic/bitwise and arithmetic-on-string is delegated to a
string-library metamethod. 5.3 instead coerces numeric strings **in the core**
for both arithmetic and bitwise, and emits the older, shorter error wording from
the core path (with varinfo).

---

## Category 1 — string-in-bitwise coercion (clear-cut)

5.3 coerces a numeric string operand to an integer in the core bitwise ops
(`& | ~ << >>` and unary `~`). We error instead.

| Code (5.3) | ref output | our output |
|---|---|---|
| `print("3" & 5)` | `1` | error "perform bitwise operation on a string value (constant '3')" |
| `print("0xff" \| 0)` | `255` | error |
| `print(~"5")` | `-6` | error |
| `print("8" >> "1")` | `4` | error |
| `print("8" << 1)` | `16` | error |
| `print(5 & "3")` | `1` | error |
| `print(" 0x10 " & 255)` | `16` | error |
| `print("3.0" & 1)` | `1` | error |

Whitespace-padded and signed/hex strings coerce too (official `bitwise.lua`
lines 53-56):

- `print("0xffffffffffffffff" \| 0)` → `-1` (hex wraps to integer, as in literals)
- `print("0xfffffffffffffffe" & "-1")` → `-2`
- `print(" \t-0xfffffffffffffffe\n\t" & "-1")` → `2`
- `print("   \n  -45  \t " >> "  -2  ")` → `-180`

### Boundary cases that MUST still error in 5.3 (verified vs ref):

- `"3.5" & 1` → `number has no integer representation` (string parses to a
  non-integral float).
- `"0xffffffffffffffff.0" | 0` (inside pcall) → `number has no integer
  representation`. **NB the wording**: a numeric-but-non-integral string yields
  the "no integer representation" message, NOT "perform bitwise operation".
- `"abc" & 1` → `attempt to perform bitwise operation on a string value` (no
  `(constant ...)` suffix in the ref for this command-line constant; see note
  on varinfo below).

### Implementation location

The bitwise fast paths in `crates/lua-vm/src/vm.rs` (`bitwise_op_rr`,
`bitwise_op_k`, `bitwise_shift_rr`, and the inline `OP_BNOT`/`OP_SHRI`/`OP_SHLI`
arms) all use `to_integer_ns(..)` / `to_integer_no_strconv()` — the no-string
variant. When that returns `None` for a string, the op does not advance `pc`,
falls through to `OP_MMBIN`, and `crate::tagmethods::try_bin_tm`
(`crates/lua-vm/src/tagmethods.rs:417`) raises the bitwise error because no
`__band`/etc. metamethod is installed on strings.

The matching string-converting helper **already exists**:
`LuaValueExt::to_integer_with_strconv` (`crates/lua-vm/src/state.rs:453`), which
routes through `crate::object::str2num`. `str2num` already accepts leading/
trailing whitespace, hex, signs, and decimal/hex-float forms, and
`to_integer_no_strconv` already returns `None` for non-integral floats — so the
"3.5 errors / 3.0 succeeds / 0xff..ff wraps" boundary falls out for free.

**Two viable shapes (decide in the FIX packet):**

1. **In `try_bin_tm` (cold path), 5.3-only:** when the metamethod lookup fails
   for a bitwise event and an operand is a `Str`, retry with
   `to_integer_with_strconv`; if both convert, compute the op and write `res`;
   only then fall to the error. Keeps the hot path untouched; one version gate
   in one cold function. **Recommended.**
2. **In the bitwise fast-path helpers:** gate `to_integer_ns` →
   `to_integer_with_strconv` on `lua_version == V53` (and V51/V52 if those
   paths are ever exercised). Touches the hot loop; more call sites.

`lua_version` is reachable as `state.global().lua_version`
(`crates/lua-vm/src/state.rs:1003`).

**Risk: clear-cut.** Pure addition behind a version gate; 5.4/5.5 already match
their refs and the helper + boundary semantics already exist.

## Category 2 — arith-on-non-coercible-string error wording (clear-cut wording, medium-care plumbing)

When a string operand cannot be coerced, 5.3 and 5.4 produce **different
messages**, and we currently emit the **5.4** message on the 5.3 path.

| Code | 5.3 ref | 5.4 ref (we match this) | our 5.3 output (WRONG) |
|---|---|---|---|
| `"abc" + 1` | `attempt to perform arithmetic on a string value` | `attempt to add a 'string' with a 'number'` | `attempt to add a 'string' with a 'number'` |
| `"abc" * 2` | `...arithmetic on a string value` | `attempt to mul a 'string' with a 'number'` | `...mul a 'string'...` |
| `-"x"` | `...arithmetic on a string value (constant 'x')` | `attempt to unm a 'string' with a 'string'` | `...unm a 'string'...` |
| `local x="a"; x+1` | `...arithmetic on a string value (local 'x')` | `attempt to add...` | `attempt to add...` |
| `aaa="z"; aaa+1` | `...arithmetic on a string value (global 'aaa')` | `attempt to add...` | `attempt to add...` |

Two distinct deltas in one:

1. **Message text.** 5.3 = `attempt to perform arithmetic on a <type> value`
   (the core `op_int_error`/"perform arithmetic on" wording already present at
   `crates/lua-vm/src/tagmethods.rs:470` and `state.rs:2783`). 5.4 = the
   string-library `trymt` wording `attempt to <op> a '<t>' with a '<t>'`
   (`crates/lua-stdlib/src/string_lib.rs:511-518`).
2. **Varinfo.** The 5.3 core path appends `(local 'x')`/`(global 'aaa')`/
   `(constant 'x')`; the 5.4 string-lib `trymt` path has **no** varinfo and adds
   a spurious `[C]: in metamethod 'add'` traceback frame.

So the 5.3 message is not a string swap inside `trymt` — it must come from the
**core** arith error path (which already produces both the right text and the
varinfo), i.e. on the 5.3 path the string-lib arithmetic metamethods must not be
the thing that raises.

### What already matches (no action)

- Non-string operands already use the core path and MATCH:
  `1 + {}`, `t.f + 1` (nil, field 'f'), `nil + 1`, `{} .. "x"` — all MATCH 5.3.
- Successful 5.3 string arith coercion already MATCHES:
  `"3" + 2`, `math.type("1"+"2")` → `integer`, `math.type("1.0"+"2")` → `float`.
  (The float-promotion-on-5.3 logic is the block at
  `crates/lua-stdlib/src/string_lib.rs:468-490`.)

### Implementation location & shape

The arith opcodes route a failed fast path through `OP_MMBIN` →
`try_bin_tm` → `call_bin_tm`, which finds `__add`/`__sub`/… **installed on the
string metatable** (`crates/lua-stdlib/src/string_lib.rs:2625-…`, the
`string_lib_arith` table) and calls the string-lib `arith`/`trymt`
(`string_lib.rs:520`, `:497`). That delegation is the 5.4 model.

For 5.3, the core must own arithmetic string coercion. Two shapes:

1. **Make the core arith path coerce strings on 5.3, and don't let the string
   metamethod own the failure message.** Concretely: `raw_arith`
   (`crates/lua-vm/src/object.rs:130`) currently uses
   `to_number_no_strconv()`/integer-only matching. On the 5.3 path it should use
   `to_number_with_strconv()` (success cases — though most already work via the
   metamethod) AND, on failure with a string operand, raise the core
   `op_int_error` "perform arithmetic on" with varinfo instead of dispatching to
   the string metamethod. The cleanest seam is in `try_bin_tm`: on 5.3, for
   arith events, **skip** the string-installed metamethod and go straight to the
   core error formatter when conversion fails (mirrors how bitwise already errors
   in-core).
2. Alternatively, fix the wording inside the string-lib `trymt` to be
   version-gated AND thread varinfo into it. **Not recommended** — `trymt` runs
   as a C function with no access to the calling bytecode's operand registers, so
   it cannot produce `(local 'x')`/`(global 'aaa')`. The varinfo only exists at
   the VM `try_bin_tm` call site (which has `p1_idx`/`p2_idx`). This is the
   decisive reason the fix belongs in the core, not in `string_lib.rs`.

**Coordinate with Category 1**: both are "on 5.3, the core (not the string
metamethod) owns coercion + the failure message for string operands in
arith/bitwise." A single `try_bin_tm` 5.3 branch can cover both: try
string→number/integer coercion; on success compute; on failure raise the core
`op_int_error` (arith) / int-rep-or-bitwise error (bitwise) with varinfo.

**Risk: clear-cut on the target wording; medium-care on plumbing** because it
reshapes which layer raises arith-on-string and must not regress 5.4/5.5 (where
the string-metamethod model and its `attempt to add...` wording are correct and
already match the refs).

## Cross-version non-regression (verified)

On **5.4** and **5.5**, both categories already match the reference:

- `5.4 / 5.5: "3" & 5` → both ref and ours: `attempt to perform bitwise
  operation on a string value (constant '3')`. (5.4/5.5 do NOT coerce strings in
  bitwise — correct.)
- `5.4: "abc" + 1` → both ref and ours: `attempt to add a 'string' with a
  'number'`. (Correct 5.4 wording.)

Any fix MUST stay behind a `lua_version == V53` gate (also V51/V52 if those
paths are reachable) and leave the 5.4/5.5 metamethod model untouched.

### Known pre-existing oracle noise (NOT part of this packet)

Every error on every version differs from the C ref by a trailing
`\t[C]: in ?` traceback frame (reproduced with `error("boom")` on 5.3/5.4 and
`nil+1`). Our top-level chunk omits that final C frame. This is universal,
pre-existing, and unrelated to coercion. The CI harness below sidesteps it
entirely by matching message **substrings** inside an in-Lua `pcall`.

## CI assertions to add (`crates/lua-rs-runtime/tests/multiversion_oracle.rs`)

The harness there is `eq(version, code, expected)` and
`err_contains(version, code, needle)`, run via in-Lua `load`+`pcall` (so
tracebacks are not compared — substring match only). `LuaVersion::{V53,V54,V55}`.

```rust
// Category 1: 5.3 coerces numeric strings in core bitwise ops.
eq(LuaVersion::V53, r#"return "3" & 5"#, "1");
eq(LuaVersion::V53, r#"return "0xff" | 0"#, "255");
eq(LuaVersion::V53, r#"return ~"5""#, "-6");
eq(LuaVersion::V53, r#"return "8" >> "1""#, "4");
eq(LuaVersion::V53, r#"return "8" << 1"#, "16");
eq(LuaVersion::V53, r#"return 5 & "3""#, "1");
eq(LuaVersion::V53, r#"return " 0x10 " & 255"#, "16");
eq(LuaVersion::V53, r#"return "3.0" & 1"#, "1");
eq(LuaVersion::V53, r#"return "0xffffffffffffffff" | 0"#, "-1");
eq(LuaVersion::V53, r#"return "0xfffffffffffffffe" & "-1""#, "-2");
eq(LuaVersion::V53, r#"return "   \n  -45  \t " >> "  -2  ""#, "-180");
// Category 1 boundaries that MUST still error on 5.3:
err_contains(LuaVersion::V53, r#"return "3.5" & 1"#, "no integer representation");
err_contains(LuaVersion::V53, r#"return "0xffffffffffffffff.0" | 0"#, "no integer representation");
err_contains(LuaVersion::V53, r#"return "abc" & 1"#, "perform bitwise operation on a string value");

// Category 2: 5.3 arith-on-non-coercible-string wording + varinfo.
err_contains(LuaVersion::V53, r#"return "abc" + 1"#, "attempt to perform arithmetic on a string value");
err_contains(LuaVersion::V53, r#"return "abc" * 2"#, "attempt to perform arithmetic on a string value");
err_contains(LuaVersion::V53, r#"return -"x""#, "attempt to perform arithmetic on a string value");
err_contains(LuaVersion::V53, r#"local x="a"; return x+1"#, "(local 'x')");
err_contains(LuaVersion::V53, r#"aaa="z"; return aaa+1"#, "(global 'aaa')");

// Cross-version non-regression: 5.4/5.5 do NOT coerce strings in bitwise,
// and keep the 5.4 string-metamethod arith wording.
err_contains(LuaVersion::V54, r#"return "3" & 5"#, "perform bitwise operation on a string value");
err_contains(LuaVersion::V55, r#"return "3" & 5"#, "perform bitwise operation on a string value");
err_contains(LuaVersion::V54, r#"return "abc" + 1"#, "attempt to add a 'string' with a 'number'");
// 5.3 successful arith coercion (already passing today — guard against regression):
eq(LuaVersion::V53, r#"return math.type("1"+"2")"#, "integer");
eq(LuaVersion::V53, r#"return math.type("1.0"+"2")"#, "float");
```

> Note on `(global 'aaa')` / `(local 'x')`: these varinfo suffixes come from the
> VM `try_bin_tm` call site (`p1_idx`/`p2_idx`), not from `string_lib::trymt`,
> which is the reason Category 2 must be fixed in the core. If the FIX lands in
> `try_bin_tm`, both the message text and the varinfo come for free from the
> existing `crate::debug::op_int_error` path.
