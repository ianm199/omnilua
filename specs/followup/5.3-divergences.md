# Lua 5.3 official-suite sweep: divergences vs lua5.3.6

Oracle: unmodified `make macosx` `/tmp/lua-refs/bin/lua5.3.6`.
Method: each `/tmp/lua-refs/lua-5.3.4-tests/*.lua` run from the test dir through
`LUA_RS_VERSION=5.3 lua-rs` (absolute path) and `lua5.3.6`, both with preamble
`_soft=true; _port=true; _nomsg=true; _U=false; arg=arg or {}`. Output + exit code
normalized (addresses, prog-name) and diffed. Every reported divergence reproduced
with `specs/oracle/diff_one.sh 5.3 '<code>'` or a direct minimal probe.

## Cross-cutting divergence (affects every error-raising file)

**Trailing `[C]: in ?` traceback frame.** Every uncaught top-level error: ref ends
the traceback with an extra `\t[C]: in ?` frame (the C `docall`/`pmain` entry); ours
omits it. Reproduced with `diff_one.sh 5.3 'error("boom")'`. This does NOT fail the
suite's own `string.find(msg, ...)` checks (they match message text only), but it
does break byte-identical comparison and is the *sole* reason math.lua's category is
"traceback". It is a single shared-core fix worth one line in the traceback builder.

## Parity estimate

11/27 byte-identical (+ sort = 12 effectively, 44%). Of the 15 real DIFFs, several
share a single root, so the distinct defect count is ~9:

1. string→number coercion in bitwise/shift ops (bitwise + constructs)
2. arithmetic-on-string error wording "perform arithmetic on a string value (local 'x')"
3. trailing `[C]: in ?` traceback frame (every error path; cosmetic but universal)
4. `\u{...}` upper-bound: accept up to 0x7FFFFFFF, error "UTF-8 value too large" above
5. `print` must call global `tostring` (error if nil) instead of internal formatting
6. `string.unpack` `c0`/fixed-size bounds checking ("initial position out of string")
7. `string.format` width/precision-too-long wording; `utf8.offset` "out of range" + fn name
8. label scoping: same name legal in disjoint/nested blocks (goto)
9. GC `__gc` finalizer error propagation + `__le`-from-`__lt` (latter = known 5.4 debt)

Note `__le`-derived-from-`__lt` is flagged in `specs/oracle/CONTRACT.md` as pre-existing
5.4 port debt (shared core, affects 5.4 too), so it is tracked separately, not 5.3-only.

## Error-wording follow-up: LANDED vs DEFERRED (mv-5.3-finish)

**LANDED** (clear-cut, matched vs lua5.3.6 exactly; 5.4/5.5 unchanged; CI in
`crates/lua-rs-runtime/tests/multiversion_oracle.rs`):

- **#2 arith-on-string wording.** 5.3 now raises `attempt to perform arithmetic
  on a <type> value (<varinfo>)` from the core path (with `(local 'x')` /
  `(global 'aaa')` / `(constant 'x')` varinfo, blaming the non-number operand),
  instead of the 5.4 string-metamethod `attempt to <op> a 'X' with a 'Y'`. Fix is
  a 5.3-only guard at the top of `try_bin_tm` (`crates/lua-vm/src/tagmethods.rs`)
  that, for arith events with a string operand, raises the core error when the
  op cannot coerce both operands to numbers (covers errors.lua:100-103). The
  coercible success path (`"3"+2`) still flows through the string metamethod, so
  5.3 float-promotion is preserved. 5.4/5.5 keep `attempt to add a 'string' with
  a 'number'`.
- **NEW: `for`-loop bound wording.** 5.3 (and 5.1/5.2) raise `'for' <what> must be
  a number` (what = `limit`/`initial value`/`step`); 5.4/5.5 use `bad 'for'
  <what> (number expected, got <type>)`. Version-gated in
  `crate::debug::for_error` (`crates/lua-vm/src/debug.rs`). This was the second
  errors.lua blocker (errors.lua:287, `for i=1,'a'`).

**DEFERRED** (not a clean 5.3-only wording swap — touches shared-core argerror /
location-prefix plumbing that is *also* wrong on 5.4/5.5, so out of scope for a
5.3 wording-only packet):

- **#7 `string.format` width/precision-too-long** (strings.lua:298). Ours emits
  `invalid conversion specification` with **no `(command line):N:` prefix** on
  *both* 5.3 and 5.4. 5.3 wants `invalid format (width or precision too long)`;
  5.4 wants `invalid conversion specification: '%100.3d'`. Two separate text
  deltas plus a missing location prefix → needs the format-spec error path
  reworked across versions, not a gate.
- **#7 `utf8.offset` OOB** (utf8.lua:117). Ours emits `bad argument #3 (position
  out of bounds)` — **missing the `to 'offset'` function name and the location
  prefix** on both 5.3 and 5.4. 5.3 wants `bad argument #3 to 'offset' (position
  out of range)`; 5.4 wants `...(position out of bounds)`. The missing fn-name +
  prefix is a cross-version `argerror` plumbing gap; only the range/bounds suffix
  is the 5.3 delta. Defer until the argerror fn-name plumbing is fixed for all
  versions.
- **errors.lua:298+ call-line attribution.** A call expression split across
  lines reports the wrong source line (ours: line 2; 5.3 ref: line 1). **5.4 also
  reports line 2** → pre-existing shared-core line-info issue, not 5.3-specific
  and not error-*wording*. Out of scope.
- **#3 trailing `[C]: in ?` frame.** Universal across all versions and all error
  paths; documented pre-existing noise, sidestepped by substring-matching CI.
