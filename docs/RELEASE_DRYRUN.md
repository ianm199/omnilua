### Why 11 "fails" are not failures

Pre-first-publish, the new crate names (and the bumped 0.1.0 versions of the
renamed-but-existing internal crates) are not yet on crates.io — the registry only
has `…@0.0.33` and below. So any crate whose manifest requires `lua-X = "^0.1.0"`
cannot resolve that sibling **until the sibling is published**. Each such crate
fails with exactly one line:

```
failed to select a version for the requirement `lua-gc = "^0.1.0"`
candidate versions found which didn't match: 0.0.33, 0.0.32, 0.0.31, ...
```

This is a *dependency-ordering* artifact, reached only **after** manifest
validation and tarball assembly succeed. We proved the packaging itself is sound
two independent ways:

1. `cargo package -p <crate> --list` succeeds for all 13 crates and the tarball
   contents are clean (see per-crate table below) — no `reference/` C sources, no
   `testes/`, no fixtures, no stray junk.
2. Every non-clean log contains exactly **one** "failed to select a version" line
   and **zero** packaging-error lines (no license / description / readme / "file
   too large" / manifest-key errors).

The real release resolves this automatically by publishing **leaves first**: once
`lua-gc@0.1.0` is on the index, `lua-types` resolves; once `lua-types` is up,
`lua-vm`/`lua-stdlib`/… resolve; and so on up to `omnilua-cli`. The CI publish job
(`.github/workflows/release.yml` → `publish-crates`) already iterates the crates in
exactly this order and is idempotent (skips any crate already on the index at the
target version), so a re-run after a partial failure is safe.

## The exact ordered publish command list (the real release)

CI does this automatically on tag push. Manual fallback (run from repo root, in
order; each waits for the index to carry the previous crate before it resolves):

```bash
for c in \
  lua-gc lua-types lua-vm lua-code lua-coro lua-lex lua-parse \
  lua-rs-lfs lua-stdlib lua-rs-derive omnilua lua-rs-hlua-shim omnilua-cli; do
  cargo publish -p "$c"          # crates.io can lag a few seconds between crates
done
```

(The CI loop additionally skips any crate already on the index at the target
version, so a re-run after a partial failure is safe. If running by hand and a
later crate fails to resolve a just-published sibling, wait ~30 s for index
propagation and re-run from the failing crate.)

## Legacy pointer-release plan (old names → omnilua)

**Out of scope for the 0.1.0 tag; entirely the user's call to trigger later.** The
old names are currently on the registries at 0.0.33:

- crates.io: `lua-rs-runtime@0.0.33`, `lua-cli@0.0.33`
- npm: `lua-rs-wasm@0.0.33`

A courtesy **0.0.34 README-only pointer release** of each tells existing dependents
where the project moved. The exact mechanics are documented in RELEASING.md
§"Legacy crates" (expanded by this T3 branch). **Key subtlety:** the live workspace
no longer contains crates named `lua-rs-runtime` / `lua-cli` (they were renamed to
`omnilua` / `omnilua-cli`), so the pointer releases **cannot** be cut from `main` —
they must be published from a throwaway scratch checkout of the last old-name
commit (or a minimal stub crate) with the version bumped to 0.0.34 and the README
replaced by the redirect notice. See RELEASING.md for the step list. These ship no
code and are sequenced **after** the 0.1.0 publish (so the redirect points at a
home that already exists).
