# Lua 5.1 numbers + PRNG — design (oracle = lua5.1.5)

Area: the float-only number model and the `math.random`/`randomseed` PRNG
contract for `LuaVersion::V51`. READ-ONLY design pass. Oracle =
`/tmp/lua-refs/bin/lua5.1.5` (cross-checked vs lua5.4.7). Every row below was
reproduced live; the lua-rs column is from a scratch `is_supported(V51)=true`
build of `lua-cli` (reverted — `git diff` on `version.rs` is clean).

The CLI already parses `LUA_RS_VERSION=5.1` → `V51`
(`crates/lua-cli/src/main.rs:849`), and `V51.number_model()` already returns
`FloatOnly` (`crates/lua-types/src/version.rs:54`). The only construction gate is
`is_supported()` (`version.rs:63`), which still excludes `V51`.

## Headline: the number model is ALREADY DONE under V51 (inherited from 5.2)

The float-only behavior shares the exact machinery validated for V52. Every
numeric probe below **already matches lua5.1.5 byte-for-byte** in the scratch V51
build, because the float-only path is keyed on `number_model() == FloatOnly`,
which V51 and V52 share. There is **no new number work** for 5.1 beyond what 5.2
landed. Confirmed-correct-under-V51:

| Probe | lua5.1.5 | lua-rs V51 (scratch) | status |
|---|---|---|---|
| `print(3.0, 3, 3.5, 5)` | `3 3 3.5 5` | identical | OK — no forced `.0` |
| `print(10/2)` | `5` | `5` | OK |
| `print(2^2)` | `4` | `4` | OK (`^` always float) |
| `print(7 % 3)` | `1` | `1` | OK (`%` always float, fmod) |
| `print(0.1, 0.2, 0.1+0.2)` | `0.1 0.2 0.3` | identical | OK (`%.14g`) |
| `print(1e15, 123456789012345)` | `1e+15 1.2345678901234e+14` | identical | OK |
| `print(tostring(1/3))` | `0.33333333333333` | identical | OK (`%.14g`, 14 sig digits) |

The "never construct `Int`" invariant is therefore already satisfied for every
**observable** numeric surface under FloatOnly — the gate-based (no `LuaValue`
fork) approach is **sufficient for 5.1 numbers**, exactly as it was for 5.2. The
only place V51 still pushes `LuaValue::Int` is inside `math.random`/`randomseed`
(see below); those produce integer-valued results that print identically, so it
is not *observable* today, but it should be flipped to `Float` under FloatOnly
for hygiene and to satisfy a future forbidden-pattern hook.

## 5.1-specific: the PRNG contract (NOT the number model)

The `math.random`/`randomseed` **bodies are the unported part**. They currently
carry only the modern 5.3/5.4/5.5 contract (`crates/lua-stdlib/src/math_lib.rs`,
`math_random` line 573, `math_randomseed` line 633), gated by an `is_v53` flag
only. None of the four 5.1 contract divergences below are handled. These are real
gate-relevant failures, not RNG-sequence noise.

| Probe | lua5.1.5 | lua-rs V51 (scratch) | divergence |
|---|---|---|---|
| `math.random(0)` | error: `bad argument #1 to '?' (interval is empty)` | returns full-range integer (3247...) | **5.4 `random(0)`=full-range int special-case is wrong for 5.1** |
| `math.random(5, 2)` | error: `bad argument #2 to '?' (interval is empty)` | `#1 to 'math.random'` | wrong arg index (`#1` vs `#2`) + wrong name |
| `math.random(1,2,3)` | error: `wrong number of arguments` | identical | OK |
| `math.randomseed()` (no arg) | error: `bad argument #1 to '?' (number expected, got no value)` | auto-seeds, returns 2 seeds | **5.1 requires the seed arg; no auto-seed** |
| `math.randomseed(42)` | returns **nothing** (0 results) | returns 2 seed words | **5.1 randomseed returns void** |
| `select('#', math.randomseed(5))` | `0` | `2` | (same root as above) |
| `pcall(math.randomseed, "x")` | `#1 to '?' (number expected, got string)` | `#1 to 'math.randomseed' (...)` | name only (`?` vs name) |

What is already **correct** about the PRNG under V51 (the *contract*, range, and
types — the parts that are NOT host-rand()-dependent):

| Probe | lua5.1.5 | lua-rs V51 | status |
|---|---|---|---|
| `math.random()` ∈ [0,1), `type`=number | true / number | true / number | OK |
| `math.random(10)` ∈ [1,10], integer-valued, `type`=number | true | true | OK |
| `math.random(5,8)` ∈ [5,8] | true | true | OK |
| `pcall(math.randomseed, 42)` accepts a number | ok | ok | OK |

### The documented divergence (allowed): the random SEQUENCE

5.1 `math.random`/`randomseed` use C `rand()`/`srand()` (`lmathlib.c` 5.1.5:
`l_rand`/`RANDOM_MAX`). The exact byte sequence for a given seed is
**host/libc/platform-dependent** and cannot be portably bit-matched in Rust.
Reproduced: `math.randomseed(1); string.format("%.4f", math.random())` →
`0.0000` on the macOS oracle vs `0.8156` on lua-rs (xoshiro). **This is a KNOWN
DOCUMENTED divergence** — the same class as the existing `math.lua` random-seed
exception. The gate must NOT compare the seeded sequence. What the gate MUST
enforce is the contract: `random()` ∈ [0,1) float; `random(n)` ∈ [1,n]
integer-valued number; `random(m,n)` ∈ [m,n]; the four argument-error/return
shapes above. Do not attempt to reuse any 5.4 xoshiro "parity" claim for 5.1.

## A broader 5.1 error-message gap (found while probing PRNG)

5.1 reports the function name as `'?'` in **every** `bad argument` error when the
builtin is called **indirectly** (e.g. `pcall(math.floor, "z")` →
`bad argument #1 to '?' (...)`), because 5.1's `luaL_argerror`/`getfuncname`
cannot recover a name through a non-syntactic call. lua-rs reports the qualified
name `'math.floor'` in that case. This is **systematic across the whole 5.1
stdlib**, not PRNG-specific.

Crucial nuance: for a **syntactic** call (`math.floor("z")` written literally),
5.1 recovers the **short** field name `'floor'` (not `'math.floor'`), and lua-rs
**already matches this exactly**. So the gap is narrow but pervasive: only the
*indirect-call* path differs (`'?'` in 5.1 vs the qualified name in lua-rs).
Fixing it faithfully is a debug-name-recovery change (likely architectural) and
should be tracked separately from numbers/PRNG; many error-message tests that
call builtins via `pcall(fn, badarg)` will diff on this. Flagged here, not in
scope for the numbers/PRNG body fix.

## Misc contract probes (all reproduced)

| Item | lua5.1.5 | note for V51 impl |
|---|---|---|
| `arg` table from a script | `table`, `arg[0]`=script path, `arg[1..n]`=args, **`arg.n`=nil**, `#arg`=n | 5.1.5 does NOT set `arg.n`; only `#arg` and positive indices. Do not fabricate `arg.n`. |
| `arg` under `-e` (no script) | `nil` | no `arg` table when there is no main script |

| `load("return 1")` | error: `function expected, got string` | **5.1 `load` takes a reader FUNCTION only**; string loading is `loadstring`'s job |
| `load(reader_fn)` | works (concatenates reader chunks) | reader-function form must work |
| `loadstring("@@@")` | `nil  "...: unexpected symbol near '@'"` | error returned as second value, not raised |

## Impl seams (where the V51-specific work lands)

1. **`crates/lua-stdlib/src/math_lib.rs` `math_random` (line 573)** — add a
   `number_model()==FloatOnly` (or `==V51`) branch:
   - remove the `random(0)` full-range special case for V51 → fall through to the
     `low > up` empty-interval error;
   - the two-arg empty-interval error must report arg index **2** under V51
     (`arg_error_impl(state, 2, ...)`), not 1;
   - push the result as `LuaValue::Float` (integer-valued) under FloatOnly, not
     `Int`, to honor the never-construct-Int rule (currently invisible but a
     latent hook violation).
2. **`math_randomseed` (line 633)** — under V51: require the seed argument
   (`check_integer(1)`; error `number expected, got no value` when absent — do
   NOT auto-seed), and return **0** results (the seed-word pushes at
   `apply_set_seed`/`math_lib.rs:688-689` must be suppressed under V51). Modern
   versions keep the 2-result behavior.
3. **Error function-name = `'?'` for indirect calls** — systematic 5.1 stdlib
   gap; track separately (debug-name recovery), not part of the PRNG body fix.
4. **`xpcall` arity** — drop extra-arg forwarding under V51 (stdlib `xpcall`
   body; gate on version). 5.1 `xpcall(f, h)` only.
5. **`load` vs `loadstring`** — under V51, `load` accepts a reader function only
   (error on string); `loadstring` is the string loader. Roster + body gate.
6. **`arg` table shape** — populate `arg[0]`, `arg[1..n]`, `#arg`; do NOT set
   `arg.n` under V51.

PRNG sequence parity is the one allowed documented divergence; everything else
above is a contract that must match and is achievable with version gates on the
existing bodies. **No deeper change (no second VM, no `LuaValue` fork) is forced
for the numbers/PRNG area** — the gate-based Option-B-aligned approach is
sufficient here. The only architectural item surfaced is the orthogonal
indirect-call `'?'` error-name recovery, which is not a numbers/PRNG concern.
