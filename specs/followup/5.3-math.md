# Follow-up: Lua 5.3 compat-math roster (issue #19)

Status: LANDED. This file specified the deltas; the per-version partition below
was CORRECTED against the reference binaries (the binding oracle) during
implementation — see "Oracle correction".

## Oracle correction (supersedes the original 5.3-only partition)

The original draft asserted all 8 functions are 5.3-only and must be `nil` in
5.4/5.5. Probing the actual reference binaries in `/tmp/lua-refs/bin` (the
binding oracle; default `make macosx` builds) showed otherwise. The real,
verified per-version partition is:

| Function(s) | lua5.3.6 | lua5.4.7 | lua5.5.0 |
|---|---|---|---|
| atan2, cosh, sinh, tanh, pow, log10 | function | **function** | nil |
| frexp, ldexp | function | function | **function** |

Why: lua5.4.7's default luaconf.h defines `LUA_COMPAT_5_3`, whose body
`#define`s `LUA_COMPAT_MATHLIB` (luaconf.h:355) — so the full compat-math set
is in the default 5.4 build. lua5.5.0 comments the macro out
(luaconf.h:351) AND moves `frexp`/`ldexp` OUTSIDE the `#if LUA_COMPAT_MATHLIB`
block in `lmathlib.c` (lines 718/720 vs the `#if` at 730), so those two survive
into 5.5 while the other six do not.

Landed gating in `crates/lua-stdlib/src/math_lib.rs`:
- `frexp`, `ldexp` → version-agnostic `MATHLIB_FUNCS` (present in 5.3/5.4/5.5).
- `atan2`(=`math_atan` alias), `cosh`, `sinh`, `tanh`, `pow`, `log10` → gated
  to `V53 | V54` in `luaopen_math`; absent in 5.5.

All three versions now MATCH their oracle on the 8-function presence probe and
on values/edge-cases (incl. subnormal `ldexp(1.0,-1074)` and `frexp(5e-324)`).

Status: DISCOVER phase, read-only. This file specifies the exact deltas to land.

## Summary

lua5.3.6 is built with `LUA_COMPAT_MATHLIB` ON (binding per the oracle
CONTRACT). It therefore exposes 8 deprecated math functions that lua-rs's
5.3 backend is currently **missing** (all return `nil`):

    math.atan2  math.cosh  math.sinh  math.tanh
    math.pow    math.log10 math.ldexp math.frexp

All 8 confirmed `nil` under `LUA_RS_VERSION=5.3` and `function` on
`/tmp/lua-refs/bin/lua5.3.6`. These are 5.3-only: they must NOT appear in 5.4
or 5.5 (lua5.4.7 / lua5.5.0 do not register them; `LUA_COMPAT_MATHLIB` was
dropped). One-arg/two-arg `math.log` already matches (MATCH on every probe) —
no change needed there.

Total divergences: **8 missing functions** (one category: the compat-math
roster). Plus 1 unrelated shared-core note (fmod-zero error text, below).

## Exact C source (reference: /tmp/lua-refs/lua-5.3.6/src/lmathlib.c, lines 304-389)

Each is a thin wrapper over the C math.h function pushing a float, EXCEPT:

- `atan2` is registered as an **alias for the existing `math_atan`**
  (`{"atan2", math_atan}`). lua-rs `math_atan` already accepts the two-arg
  `atan(y, x)` form (verified: `math.atan(1,1)` == `math.atan2(1,1)` ==
  0.78539816339745). So `atan2` needs NO new function — just register the
  existing `math_atan` under the name `b"atan2"`.
- `frexp(x)` returns **two** values: float mantissa, then **integer**
  exponent (`lua_pushinteger`). `math.frexp(8.0)` -> `0.5  4` (m is float,
  e is integer). `math.frexp(0.0)` -> `0.0  0`.
- `ldexp(x, e)`: e is `(int)luaL_checkinteger(L,2)` (integer arg), result is
  float. `math.ldexp(0.5,3)` -> `4.0`; `math.ldexp(1.0,-1)` -> `0.5`.
- `pow(x,y)`, `cosh`, `sinh`, `tanh`, `log10`: each `luaL_checknumber` its
  arg(s) and push a float (`lua_pushnumber`). `pow` takes 2 numbers.

## File to edit

`crates/lua-stdlib/src/math_lib.rs`

Mirror the existing bit32/warn per-version roster gate
(`crates/lua-stdlib/src/init.rs` lines 94-101, which registers `bit32` only
when `state.global().lua_version == LuaVersion::V53`).

Concretely in `math_lib.rs`:

1. Add 6 new wrapper fns: `math_cosh`, `math_sinh`, `math_tanh`, `math_pow`
   (2-arg), `math_log10`, `math_ldexp` (arg2 = `check_integer`, cast to i32
   range like C `(int)`), `math_frexp` (push mantissa float + exponent as
   `LuaValue::Int`). No new fn for `atan2` — reuse `math_atan`.
   - frexp: use `x.frexp()`-equivalent. Rust std has no `frexp`; replicate via
     the C identity, or compute mantissa/exponent from bit manipulation, or
     pull from `libm`. Match `frexp(0.0)=(0.0,0)`. Validate against oracle.
   - ldexp: `x * 2f64.powi(e as i32)` matches C `ldexp`; verify edge exps.
2. In `luaopen_math`, AFTER the base roster is registered and only when
   `matches!(state.global().lua_version, lua_types::LuaVersion::V53)`,
   register the 8 names:
     atan2 -> math_atan, cosh, sinh, tanh, pow, frexp, ldexp, log10.
   Do NOT add them to the version-agnostic `MATHLIB_FUNCS` table (that would
   leak them into 5.4/5.5 and regress the 5.4 gate).
3. Update the PORT STATUS trailer note (currently says compat fns omitted) to
   reflect the 5.3-gated registration.

## Shared-core edges probed (NOT math-roster scope) — all MATCH except one

MATCH on 5.3 vs lua5.3.6: `math.modf(3.7)` (-> `3  0.7`, int part is integer
type), `math.modf(1/0)`, `math.modf(2^63)`, `math.fmod(5,3)` (int 2),
`math.fmod(-5,3)`, `math.fmod(5,-3)`, `math.fmod(5.0,0)` (nan), `-math.huge`
(-inf), `0/0` (nan/-nan), `math.maxinteger+1==math.mininteger`,
`math.tointeger`, `math.floor/ceil` types, `math.abs(math.mininteger)`.

ONE divergence, shared-core (affects 5.3 AND 5.4 equally — pre-existing,
NOT introduced by and NOT closed by this math task):
`math.fmod(5,0)` (integer zero divisor)
  ref : `bad argument #2 to 'fmod' (zero)`
  ours: `bad argument #2 (zero)`   <- missing `to 'fmod'` and source-position
The `luaL_argerror`/raise path used by `math_fmod`'s integer-zero check does
not thread the function name (unlike `check_number`, which does). Flag as a
separate shared-core arg-error-naming follow-up; out of scope here.
