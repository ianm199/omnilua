The error message above shows ONLY the C-loader (.so) attempted paths. That
means the Lua loader (.lua paths) was tried and failed silently, OR was
never tried.

Likely root causes (in order):

1. **package.path is being reset / wrong inside the running Lua program.**
   Could be locals.lua sets `package.path = X` and X doesn't include the test
   directory. Look near where it `require`s things in locals.lua.

2. **searchpath bug.** `package.searchpath` is what implements the path
   search. If it lowercases / mangles / drops template substitutions, the
   Lua-loader silently fails. Check `crates/lua-stdlib/src/loadlib.rs:searchpath`.

3. **The require finder ordering changed.** Slice 02b touched coro_lib but
   should NOT have touched loadlib. But check git log:
     git log --oneline -- crates/lua-stdlib/src/loadlib.rs

   Especially around dynlib changes today (commit 8c48cb1) — that touched
   loadlib heavily. Maybe the searcher chain regressed.

4. **`coroutine.wrap` regression**. Slice 02b removed `wrap_iter_state`. If
   any internal Lua iterator (probably NOT — gmatch isn't coroutine-based)
   used the hack, this could affect require's iteration over searchers.
   Long shot — investigate only if #1-3 ruled out.
