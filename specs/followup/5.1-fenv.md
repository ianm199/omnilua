# 5.1 fenv globals — design (Option B, reuse the modern `_ENV` machinery)

Status: design / read-only investigation. Audience: whoever implements the 5.1
backend's globals model. Oracle: `/tmp/lua-refs/bin/lua5.1.5`. Everything below
was reproduced against that binary.

Verdict up front: **Option B is sufficient for fenv.** No second VM ISA is
needed. The modern parser already threads an `_ENV` upvalue (upvalue index 0) on
every closure and resolves free names via GETTABUP/SETTABUP against it; under
V51 that upvalue 0 *is* the function environment, and `getfenv`/`setfenv`
read/write it. The one genuine gap is C-function environments
(`LUA_ENVIRONINDEX`), documented below — it does not block the Lua-side surface.

---

## 1. The mechanism (where the per-closure env lives)

Under the modern core, every Lua closure carries `upvals[0]` named `_ENV`
(`crates/lua-parse/src/lib.rs:4674-4682`, `mainfunc`; free names resolve through
it, `:2376-2382`, `:4308-4385`). At load time the main chunk's `_ENV` is bound to
a closed upval holding the global table:

```
// crates/lua-vm/src/api.rs:1801-1806
if let LuaValue::Function(LuaClosure::Lua(lcl)) = func_val {
    if !lcl.upvals.is_empty() {
        let gt = get_global_table(state);
        let uv = state.new_upval_closed(gt);
        lcl.set_upval(0, uv);     // <-- upval[0] = _ENV = the function environment
    }
}
```

So a closure's "environment" already has a concrete home: **`upvals[0]`'s closed
value**. Child closures inherit it because the parser shares the parent's `_ENV`
upvalue down into nested protos (standard upvalue capture) — this matches the
5.1 rule that a new closure inherits the creating function's environment
(verified: a closure made inside an `outer` whose env was `setfenv`'d resolves
free names in that env).

Therefore under V51:
- **Per-closure env == `upvals[0]` (the `_ENV` upval).** `getfenv`/`setfenv` of a
  *function* read/write that upval's closed value.
- **Thread global table == `state.global().globals`** (`get_global_table`). This
  is the env that **freshly `load`/`loadstring`-ed top-level chunks** receive and
  is what `getfenv(0)` returns / `setfenv(0, t)` mutates. It is *separate* from
  any individual running closure's env.

No new field is required on the closure: the env is the existing `_ENV` upval.
The thread env is the existing globals table. Option B is "wire `getfenv`/
`setfenv` to these two existing slots, and make V51's `load` seed upval 0 from the
thread global table (which it already does) rather than from a fixed `_G`."

### The seams `getfenv`/`setfenv` reach through (all already exist)

- Read/write a closure's upval-0 closed value: `set_upvalue`/`get_upvalue`
  exist and are already used by the 5.2 `load(chunk,name,mode,env)` env-binding
  path (`crates/lua-stdlib/src/base.rs:162-166`, `load_aux` calls
  `state.set_upvalue(-2, 1)` to bind a custom `_ENV`). **This is the exact prior
  art** — `setfenv(f, t)` is `load_aux`'s env-bind generalized to an arbitrary
  already-built closure.
- Resolve a stack *level* to a closure: the debug/`getinfo` activation-record
  walk (the same level→callinfo mechanism `getfenv(level)`/`setfenv(level)` and
  `getfenv()`/`getfenv(1)` need). Levels count Lua frames; `pcall` does add a
  frame (verified `getfenv(1)` inside a called fn sees that fn, not the caller).
- Thread global table read/write: `state.global().globals`
  (`crates/lua-vm/src/api.rs:1204`). `setfenv(0,t)` reassigns it; `getfenv(0)`
  returns it.

## 2. Observable semantics (all reproduced against lua5.1.5)

### getfenv
- `getfenv()`, `getfenv(0)`, `getfenv(1)` in the main chunk all return `_G`
  (`== true`). In the main chunk `getfenv(0) == getfenv(1)`.
- `getfenv(f)` (function arg) returns that function's env upval value.
- `getfenv(level)` returns the env of the function `level` frames up the running
  stack (`1` = the function calling getfenv).
- `getfenv()` (no arg) == `getfenv(1)` (the running function).
- Number levels are **floored**, not required to be integral: `getfenv(0.9)`
  behaves as level 0, `getfenv(1.9)` as level 1 (float-only world — accept any
  number, truncate toward zero).
- C functions: `getfenv(print)` returns `_G` (5.1 gives C functions the thread
  global table as their env). See the documented gap (§4).
- Errors:
  - level out of range → `bad argument #1 to '?' (invalid level)`
  - negative level → `bad argument #1 to '?' (level must be non-negative)`
  - non-number, non-function arg → `bad argument #1 to '?' (number expected, got <type>)`

### setfenv
- `setfenv(f, t)` sets a **Lua function's** env; **returns the function** `f`.
- `setfenv(level, t)` sets the env of the function at that stack level; **returns
  that function** (verified: `setfenv(1, t)` returns a `function:` value).
- After `setfenv(f, t)`, calling `f` resolves its free names in `t` (verified:
  `f` returning a free `x` yields `t.x`), and the caller's own globals are
  unaffected.
- `setfenv(0, t)` sets the **running thread's global table** (the env handed to
  subsequently-loaded top-level chunks). Critical subtlety, all verified:
  - `getfenv(0)` afterwards returns `t`; **`getfenv(1)` does NOT** — the
    currently-running closure keeps its own (pre-existing) `_ENV` upval. So
    `setfenv(0,t)` does *not* retroactively change the running closure's free-name
    resolution: a free name in the running chunk still hits the old env (verified:
    `setfenv(0, {z=99,...}); print(z)` prints `nil`, while `t.z` is `99`).
  - Functions the running closure creates *after* `setfenv(0,t)` still capture the
    running closure's env, **not** `t` (verified: a closure made after the call
    does not see `t`).
  - A chunk produced by `loadstring`/`load` *after* `setfenv(0,t)` gets `t` as its
    env (`getfenv(chunk) == t`, and `getfenv(chunk) == getfenv(0)`), **even when
    the running closure's own env was changed** by a separate `setfenv(1, cenv)`.
    I.e. loaded chunks take the **thread** env, never the loader's closure env
    (verified: `setfenv(1, cenv); loadstring("return a")` → chunk env is the thread
    env, not `cenv`).
- Errors:
  - C function / non-Lua-function object → `'setfenv' cannot change environment
    of given object`
  - bad table arg → `bad argument #2 to '?' (table expected, got <type>)` (e.g.
    `got string`, `got no value`)
  - level out of range → `bad argument #1 to '?' (invalid level)`

### The two-slot model, restated
There are **two** environment slots and `setfenv`/`getfenv` distinguish them by
the first argument:

| first arg | slot read/written |
|---|---|
| `0` | the **thread global table** (`state.global().globals`); also the default env for newly-loaded top-level chunks |
| `>=1` (level) | the **`_ENV` upval** of the Lua closure that many frames up |
| function value | that closure's **`_ENV` upval** |

This is the single most fakeable area: a naive impl that makes everything point
at `_G` passes "globals work" but fails the per-closure-≠-thread distinction.
The tests in §3 pin exactly that distinction.

## 3. Test cases (capture expected values from lua5.1.5)

For `multiversion_oracle.rs` `v51_*` tests (each via the `load`+`pcall` wrapper),
plus a curated `check.sh 5.1` battery. Expected values verified above.

1. `getfenv() == _G`, `getfenv(0) == _G`, `getfenv(1) == _G` in main → `true`.
2. Per-closure env differs from thread:
   ```lua
   local function f() return x end
   local e = setmetatable({x=42},{__index=_G})
   setfenv(f, e); return f()        --> 42
   ```
   and `x` in the caller stays `nil`.
3. `setfenv(f,e)` returns `f` (`setfenv(f,{})==f` → `true`).
4. New closure inherits creator env:
   ```lua
   local function outer() local function inner() return secret end return inner end
   local e=setmetatable({secret="hi"},{__index=_G}); setfenv(outer,e)
   local inner=outer(); return getfenv(inner)==e and inner()  --> "hi"
   ```
5. `setfenv(0,t)` thread/closure split: after `setfenv(0,t)`,
   `getfenv(0)==t` is `true` but `getfenv(1)==t` is `false`, and a free name set
   only in `t` reads back `nil` from the still-running closure.
6. Loaded chunk takes thread env:
   ```lua
   local e=setmetatable({secret="thr"},{__index=_G}); setfenv(0,e)
   local c=loadstring("return secret"); return getfenv(c)==e and c()  --> "thr"
   ```
7. Level form: `getfenv(2)` from a callee returns the caller's env;
   `setfenv(2,t)` sets the caller's env (verified the caller's free name then
   resolves in `t`).
8. Error cases (exact messages in §2): `setfenv(print,{})`, `setfenv(0,"x")`,
   `getfenv(5)`, `getfenv(-1)`, `getfenv("x")`, `setfenv(100,{})`.
9. Float level truncation: `getfenv(1.9)` behaves as level 1.

A **globals kit** (rung-2 in-memory, per CLAUDE.md fast-iteration strategy)
should exercise the §2 two-slot matrix without the oracle: build closures with
distinct `_ENV` upvals, assert getfenv/setfenv read/write the right slot, assert
the thread-vs-closure split.

## 4. Documented gap — C-function environments (`LUA_ENVIRONINDEX`)

5.1 C functions have their own environment (manual §2.9; `lua_getfenv`/
`lua_setfenv` on a C function, pseudo-index `LUA_ENVIRONINDEX`). In our engine
C/Rust builtins are not Lua closures and carry no `_ENV` upval. Observably:
`getfenv(print)` returns `_G` in 5.1.

Decision (Option B v1): **`getfenv(<C function>)` returns the thread global
table** (matches the common 5.1 case, since stdlib C functions share `_G`), and
**`setfenv(<C function>, t)` errors** with `'setfenv' cannot change environment
of given object` — which is *exactly* what stock lua5.1.5 does for objects whose
env cannot be changed (verified). The genuine gap is a host that registers a
Rust function and expects a *distinct, mutable* per-C-function environment
readable/writable via get/setfenv. **No consumer in this port needs that**
(`module`/`package.seeall`/sandbox idioms all operate on Lua closures or the
thread env). Document it as a known limitation; do **not** fabricate a fake
per-Rust-fn env slot. If a consumer ever needs it, add an optional env field to
the Rust-closure object and route get/setfenv to it — additive, not a redesign.

## 5. Impl seams (concrete)

- **Lift the gate** (`crates/lua-types/src/version.rs:63-68`): add `V51` to
  `is_supported()` — *only after* the battery passes. The runtime guard
  (`crates/lua-rs-runtime/src/lib.rs:781`) and CLI parse
  (`crates/lua-cli/src/main.rs:849`) already handle `V51` once supported.
- **getfenv/setfenv registration**: V51-gated entries in the base-lib registry
  (alongside `loadstring`/`unpack`/etc. from §2.3 of the spec). Bodies:
  - resolve arg → (thread-env | closure `_ENV` upval) per the §2 table;
  - read = clone the slot value; write = `set_upvalue`-style write to upval 0
    (reuse `load_aux`'s `set_upvalue(-2,1)` path) or reassign
    `state.global().globals` for the `0` case;
  - level→closure resolution reuses the debug activation-record walk.
- **V51 `load`/`loadstring` env seeding**: the main-chunk upval-0 bind in
  `api.rs:1803` should source the **thread global table** (it already does via
  `get_global_table`); ensure `setfenv(0,t)` updating `state.global().globals` is
  observed by subsequent loads. (5.1 `load` takes a *reader function only*;
  `loadstring` takes the string — a §2.3 roster item, orthogonal to fenv.)
- **No parser/opcode change**: the modern `_ENV`/GETTABUP path is reused
  unchanged. `OP_GETGLOBAL`/`OP_SETGLOBAL` are *not* introduced — that is Option A
  and is explicitly rejected. The 5.1 `OP_LOADNIL` range-form concern (spec §2.2)
  is a loader/codegen detail unrelated to fenv and does not arise here because we
  reuse the modern compiler, not a 5.1 bytecode loader.

---

## 6. Does any case force a deeper change?

No. Every observable fenv behavior maps onto two already-existing slots (the
closure `_ENV` upval and the thread global table) reached through already-existing
machinery (`set_upvalue`/`get_upvalue` — proven by the 5.2 `load` env arg — and
the debug level walk). The only thing Option B *cannot* express is a distinct
mutable per-C-function environment, which is the documented gap in §4 and which
no consumer in this port exercises. **Option B / gate-based approach is
sufficient for the fenv axis.**
