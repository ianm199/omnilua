# Lua 5.1 roster / syntax / metamethods — oracle-confirmed (DESIGN: roster_syntax)

Every claim below was reproduced against `/tmp/lua-refs/bin/lua5.1.5` (the pinned
oracle). Where the result contradicts the planning spec
(`specs/LUA_5_1_PORT_SPEC.md`, `specs/research/5.1-5.2-upstream.md`), the oracle
wins and the contradiction is called out explicitly. This file covers the
roster + syntax + metamethods sub-area only; the float-only core and the fenv
globals subsystem are owned by other DESIGN areas.

`_VERSION` == `"Lua 5.1"`.

## 1. Global roster — exact field dumps from the oracle

These are the COMPLETE `pairs()` dumps of each library table, so they double as
the authoritative present/absent lists. Anything not in a list is ABSENT.

**`_G` (non-table values):**
`_VERSION, assert, collectgarbage, dofile, error, gcinfo, getfenv, getmetatable,
ipairs, load, loadfile, loadstring, module, newproxy, next, pairs, pcall, print,
rawequal, rawget, rawset, require, select, setfenv, setmetatable, tonumber,
tostring, type, unpack, xpcall`
(plus the library tables: `_G, coroutine, debug, io, math, os, package, string,
table`)

- `unpack` is a **global** (no `table.unpack`).
- `getfenv`/`setfenv` present; `loadstring` present; `module` present;
  `newproxy`, `gcinfo` present.
- **ABSENT from 5.4 that you must NOT register under V51:** `rawlen` (nil),
  `math.type` (nil — see below), `bit32` (nil — it is **5.2-only**, not 5.1),
  `string.pack`/`utf8`/`warn`/`require`-searchers-rename.

**`math`:**
`abs, acos, asin, atan, atan2, ceil, cos, cosh, deg, exp, floor, fmod, frexp,
huge, ldexp, log, log10, max, min, mod, modf, pi, pow, rad, random, randomseed,
sin, sinh, sqrt, tan, tanh`
- **`math.mod` IS PRESENT** (a compat alias of `fmod`: `math.mod(7,3)==1`). The
  planning spec said `math.mod` is absent — **that is WRONG for 5.1.5**; register
  it. Also present and 5.4-absent: `math.atan2`, `math.pow`, `math.log10`,
  `math.cosh`, `math.sinh`, `math.tanh`, `math.frexp`, `math.ldexp`.
- **`math.type` ABSENT** (nil). **No `math.tointeger`, `math.maxinteger`,
  `math.mininteger`, `math.ult`.**
- **`math.log` takes ONE argument and SILENTLY IGNORES any 2nd argument** — it
  does NOT error. `math.log(8,2)` == `math.log(8,99)` == `ln(8)` ==
  `2.0794415416798`. The planning spec's claim that `math.log(8,2)` errors is
  WRONG. The body must compute `ln(x)` and ignore arg 2 (do not error, do not use
  it as a base).

**`string`:**
`byte, char, dump, find, format, gfind, gmatch, gsub, len, lower, match, rep,
reverse, sub, upper`
- `string.gfind` present (compat alias of `gmatch`); both work identically.
- `%z` pattern class still works (`string.match("a\0b","a%zb")=="a"`).

**`table`:**
`concat, foreach, foreachi, getn, insert, maxn, remove, setn, sort`
- `table.getn/foreach/foreachi/maxn` present and functional.
- `table.setn` is registered but is a **gravestone**: calling it raises
  `'setn' is obsolete` (it is `pcall`-able to false). Register it as a stub that
  raises that exact message; do not implement it.
- **No `table.unpack`, no `table.move`, no `table.pack`.**

**`os`:**
`clock, date, difftime, execute, exit, getenv, remove, rename, setlocale, time,
tmpname`
- `os.execute("exit 3")` returns the **raw `wait()` status** (`768` = `3<<8` on
  this host); `os.execute()` returns `1` (truthy = shell available). This is the
  5.1 contract (a number), NOT the 5.2 `true/nil + "exit"/code` shape.
  HOST-DEPENDENT byte; treat exact value as documented divergence, contract (a
  single number) is what must match.

**`package`:**
`config, cpath, loaded, loaders, loadlib, path, preload, seeall`
- `package.loaders` present; **`package.searchers` ABSENT** (that's the 5.2
  rename). `package.seeall` present.

**`coroutine`:**
`create, resume, running, status, wrap, yield`
- `coroutine.running()` returns **nil in the main coroutine** (confirmed).
- No `coroutine.isyieldable`, no `coroutine.close`.

**`collectgarbage` options:** `collect/stop/restart/count/step/setpause/
setstepmul`. `collectgarbage("isrunning")` ERRORS:
`bad argument #1 to '?' (invalid option 'isrunning')` — the 5.4 options
(`isrunning`, `generational`, `incremental`) must NOT be accepted under V51.

### Arity / behavior deltas to gate (not just present/absent)

- **`load(f)` takes a READER FUNCTION only.** `load("return 1")` (a string)
  errors: `bad argument #1 to 'load' (function expected, got string)`. String
  loading is `loadstring`'s job.
- **`loadstring(s [, chunkname])`** loads a string; 2nd arg is the chunk name.
- **`xpcall(f, h)` CANNOT take extra args.** `xpcall(fn, h, 1,2,3)` calls `fn`
  with ZERO args (`select("#",...)==0` inside `fn`). Do not forward extra args
  under V51.
- **`gcinfo()`** returns a number (KB count); present as a deprecated holdover.

## 2. Syntax — what must be REJECTED, with EXACT oracle messages

All from `lua5.1.5 -e '<src>'`. The byte-exact message matters for the
errors-corpus.

| Construct | Source | Oracle error (after the `file:line:` prefix) |
|---|---|---|
| `goto` stmt | `do goto done; ::done:: end` | `'=' expected near 'done'` |
| `::label::` | `::lbl::` | `unexpected symbol near ':'` |
| floor-div `//` | `print(7//2)` | `unexpected symbol near '/'` |
| bit-and `&` | `print(5 & 3)` | `')' expected near '&'` |
| bit-or `\|` | `print(5 \| 3)` | `')' expected near '\|'` |
| bit-xor `~` (binary) | `print(5 ~ 3)` | `')' expected near '~'` |
| shift `<<` | `print(1 << 4)` | `unexpected symbol near '<'` |
| `<const>` attrib | `local x <const> = 5` | `unexpected symbol near '<'` |
| `<close>` attrib | `local x <close> = nil` | `unexpected symbol near '<'` |

KEY INSIGHT — **`goto` is NOT a reserved word in 5.1.** `local goto = 5` is
valid and prints `5`. The `goto`-statement error message above is a side effect
of the parser treating `goto` as an ordinary name beginning an assignment
statement (hence `'=' expected near 'done'`). The V51 parser must therefore:
(a) NOT add `goto`/`::` to the keyword/label grammar, and (b) keep `goto` as a
usable identifier. Same logic gives the bitwise/`//` messages — those tokens
simply don't exist, so the error is whatever the surrounding grammar expects
next (often `')' expected` or `unexpected symbol`). Do not synthesize a custom
"bitwise not supported" message; reproduce the incidental parser message.

### String escapes — SUBTLE, contradicts the planning spec

- `\x41` does **NOT error** and is **NOT a hex escape**. The 5.1 lexer does not
  recognize `\x`; for an unrecognized escape it **silently drops the backslash
  and keeps the next char**. `"\x41"` → bytes `120 52 49` = literal `"x41"`.
- `\z` likewise → literal `"z"` (1 byte `122`); whitespace is NOT skipped.
- `\q` (any unknown letter) loads fine → literal `"q"`. **5.1 does NOT raise
  "invalid escape sequence".** This is the OPPOSITE of 5.2+ and of the planning
  spec's "5.1 lexer rejects `\x`" claim. The V51 lexer escape handler must, for
  any char not in the known set, emit that char literally (drop the backslash) —
  it must NOT error and must NOT treat `\x`/`\z` specially.
- Known escapes still work: `\65` (decimal) → `"A"`; `\n\t\r\a\b\f\v\\\"\'` and
  `\ddd`. Decimal escape `\999` (>255) DOES error:
  `escape sequence too large near '"'`.

### Number literals

- Hex INTEGER literals present: `0x1F`==31, `0xff`==`0XFF`==255 (lower to f64).
- Decimal exponent `1e3`==1000.
- `0x1.8p1` (hex float WITH a fractional dot) → ERROR:
  `malformed number near '.8p1'`.
- **TRAP — `0x1p4` returns 16 on this host.** 5.1's `lua_str2number` is
  `strtod`, and a C99 `strtod` accepts the `0xNpN` hex-float-no-dot form. So the
  oracle ACCEPTS `0x1p4` even though hex floats are nominally 5.2+. This is
  **host-libc-dependent**, not a portable 5.1 guarantee. RECOMMENDATION: the V51
  lexer should reject the `p`/`P` binary exponent (the spec'd 5.1 behavior); the
  fact that the macOS oracle happens to accept `0x1p4` via libc `strtod` is a
  KNOWN-DIVERGENT host artifact and must NOT be encoded as a test expectation.
  Do not build a corpus assertion on `0x1p4`.

Absent libraries to confirm-reject at use sites: `string.pack`/`string.unpack`/
`string.packsize` (nil), `utf8` (nil), `bit32` (nil).

## 3. Metamethod gates — the silent-failure traps

| Metamethod | 5.1 oracle behavior | lua-rs today |
|---|---|---|
| `__len` on a **table** | **INERT.** `#setmetatable({1,2,3},{__len=function() return 99 end})` == **3** (primitive length). | **CONSULTS `__len`** → would return 99. **WILL DIVERGE.** |
| `__len` on **newproxy userdata** | **WORKS.** `#p` == 42 via the userdata metatable. | works (userdata path already metamethod-aware) |
| `__pairs` | **IGNORED.** A `__pairs` that `error()`s never fires; `pairs(t)` iterates the raw table. | n/a — must not consult |
| `__gc` on a **table** | **INERT.** Setting `__gc` on a table metatable and collecting does NOT call it (no error, no call). | must not finalize tables under V51 |
| `__gc` on **newproxy userdata** | **WORKS.** Fires on collection. | userdata finalizer path |
| `__lt` derives `__le` | **KEPT.** With only `__lt` defined, `x<=y` works (returns true). Already gated 5.1–5.4 in lua-rs. | OK (shared) |

CONFIRMED CONTRADICTION (the spec's #1 silent-failure prediction is real):
lua-rs's `#`/length path **does** consult `__len` on tables today, so under V51
it diverges from the oracle. See seam below.

`__index`/`__newindex`/`__call`/`__concat`/`__eq`/`__metatable`/`__mode`
unchanged from the modern core. No `__name`, no `__idiv`/`__band`/etc. (5.3+).
Arithmetic metamethods are always float-dispatched (shared with 5.2 — not a
roster_syntax concern).

## 4. Implementation seams (where each gate lands in lua-rs)

- **Roster gate — `crates/lua-stdlib/src/init.rs`.** `LOADED_LIBS`
  (init.rs:58) + the existing `matches!(lua_version, V51|V52)` block
  (init.rs:94) is the right hook. Per-version registration must:
  - register under V51: global `unpack`, `loadstring`, `getfenv`/`setfenv`,
    `module`, `newproxy`, `gcinfo`, `string.gfind`, `math.{mod,pow,log10,atan2,
    cosh,sinh,tanh,frexp,ldexp}`, `table.{getn,setn(stub-raises),maxn,foreach,
    foreachi}`, `package.{seeall,loaders}`;
  - withhold under V51: `rawlen`, `math.type`, `table.{unpack,pack,move}`,
    `bit32` (note: bit32 is 5.2-only; the existing init.rs gate at line 109
    correctly limits bit32 to `V52|V53`), `package.searchers`, `string.pack`/
    `utf8`, `coroutine.{isyieldable,close}`.
  - body changes gated on version: `math.log` ignores arg 2 (no base); `load`
    rejects string arg (reader-fn only); `xpcall` forwards zero extra args;
    `collectgarbage` rejects 5.4-only options; `os.execute` returns the numeric
    status, not the 5.2 `true/nil` shape.

- **Syntax gates — the parser/lexer (`crates/lua-parse`, `crates/lua-lex`).**
  Under V51: do NOT enable `goto`/`::label::` grammar (and keep `goto` a valid
  identifier); do NOT enable `<const>`/`<close>` attrib syntax; the lexer must
  not emit floor-div/bitwise/shift tokens; the string-escape handler must drop
  the backslash on UNKNOWN escapes (emit the bare char, no error) and must NOT
  special-case `\x`/`\z`; the number scanner should reject the `p`/`P` hex-float
  exponent (host-`strtod` accepting `0x1p4` is a documented divergence — do not
  gate a test on it).

- **`#` / `__len` dispatch — `crates/lua-vm/src/state.rs:2834` `obj_len`** (also
  the `OpCode::Len` arm in `vm.rs:2229` which routes here, and the duplicate
  metatable read at `vm.rs:1297`). The `LuaValue::Table` arm currently reads
  `fast_tm_table(.., TagMethod::Len)` and calls it if present. **The V51 fix:**
  in the Table arm, when the engine version is V51, SKIP the `__len`
  consultation entirely and always return `table_length(v)`. Leave the userdata
  arm (`other =>`) unchanged so `newproxy`-userdata `__len` still works. This is
  the single highest-risk gate — guard it with an oracle corpus case
  (`#setmetatable({1,2,3},{__len=...})==3`).

- **`__gc`/`__pairs` table eligibility** — the GC finalizer-eligibility check
  (the `__gc`-on-table path) and `pairs` must be version-gated so tables are
  never finalized and `__pairs` is never consulted under V51. (GC eligibility is
  shared with the GC DESIGN area; flag the V51 table-finalizer-OFF requirement
  there.)

## 5. Sufficiency verdict for this area

The Option-B / gate-based approach is **SUFFICIENT for roster + syntax +
metamethods**. None of these cases forces a deeper (second-ISA) change:

- Roster is already data-driven in `init.rs`; it's per-entry gating + a handful
  of version-gated bodies. No module fork needed.
- Syntax rejections fall out of simply NOT enabling the 5.2+/5.4 grammar under
  V51 (plus the one lexer-escape leniency flip). No new parser needed.
- The `__len`/`__gc`/`__pairs` flips are localized branches in `obj_len` and the
  finalizer-eligibility check. Branch on version; done.

The only genuine code-behavior change (vs "just don't register") is the
`__len`-on-tables flip in `obj_len` — and it is a 2-line version branch.

## Top fixes (priority order)

1. **`obj_len` Table arm (`state.rs:2834`): under V51 skip `__len`, always use
   primitive `table_length`.** This is the confirmed #1 silent-failure; lua-rs
   diverges today.
2. **Lexer unknown-escape leniency under V51:** drop backslash, emit bare char,
   NO error; do NOT special-case `\x`/`\z`. (Opposite of 5.2+.)
3. **`math.log` body under V51: ignore the 2nd arg, never error, return ln(x).**
   (Spec was wrong; oracle silently ignores the base.)
4. **Register `math.mod` under V51** (spec said absent; oracle has it as an
   fmod alias).
5. **`table.setn` = stub that raises `'setn' is obsolete`** (registered but
   gravestone). `xpcall` drops extra args; `load` rejects string arg;
   `collectgarbage` rejects 5.4-only options under V51.
6. **Parser: keep `goto` a valid identifier**, do not enable goto/label/`<attrib>`
   grammar; reproduce the incidental parser messages (table §2), don't synthesize
   custom ones.

## Documented divergences (NOT gate failures)

- `0x1p4` accepted by the macOS oracle via libc `strtod` — host artifact; V51
  should reject the `p` exponent and we do not test the oracle's acceptance.
- `os.execute` raw-status byte (`768`) and `math.random` sequence are
  host/platform-dependent; match the contract/shape, not exact bytes.
